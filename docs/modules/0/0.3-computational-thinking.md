# Systems Mastery 0.3 — Computational Thinking

> Software engineering begins before code: with the transformation of vague problems into precise computational procedures.

---

# Objective

Computational thinking is the foundation of every implementation that will appear later in Systems Mastery.

Before algorithms, data structures, operating systems, databases, networks or distributed systems, an engineer must learn to transform a vague problem into something a machine can execute.

A computer does not understand intention.

It does not understand “problematic order”, “valid user”, “completed purchase” or “available bike” unless those ideas have been made precise.

Computational thinking is the discipline of making them precise.

---

# From Vague Problems to Precise Procedures

A requirement often begins as natural language:

> Find problematic orders.

This is not yet computable.

The first engineering task is not to write code. It is to ask what “problematic” means.

An order may be problematic because data is missing, payment failed, the product is unavailable, or the address is invalid.

Only when the vague idea becomes observable can the system act on it.

The transformation is:

```text
vague problem
→ precise conditions
→ procedure
→ result
```

This is the first mental shift of computational thinking:

> A computer does not solve vague problems. It executes precise models of those problems.

---

# Software as State Transformation

Every component can be understood as a machine that observes state, applies rules and produces a new state.

A bank account is not first a class, a table or an API.

Computationally, it is state.

```text
Balance = 500
```

A deposit is a transformation.

```text
Balance = 500
deposit 100
Balance = 600
```

The state describes how the world is now.

The procedure describes how the world may change.

This distinction will reappear everywhere: databases transform persistent state, operating systems transform resource state, network protocols transform communication state, and AI runtimes transform tensor and memory state.

---

# Valid and Invalid States

Not every possible state should be allowed.

A banking system may forbid negative balances.

An authentication system should not allow an authenticated user who does not exist.

A cinema booking system should not allow the same seat to be booked twice.

These are not implementation details. They are properties of the problem itself.

A correct system is not merely one that performs operations. It is one that prevents impossible states.

This leads to one of the most important ideas in software reasoning:

> An invariant is a property that must remain true as the system changes.

Operations may transform state, but they must preserve the invariants that make the system valid.

---

# Decomposition

Some problems are too large to reason about directly.

“Build an airport management system” is not a problem the mind can hold all at once.

The first task is to make the problem thinkable.

Decomposition does not merely divide work into steps. It separates different questions.

In a checkout system, these questions may be distinct:

```text
What is the user buying?
How much must they pay?
Where should the order be sent?
Is payment possible?
Can the order be confirmed?
```

A good decomposition separates responsibilities, inputs, failures and invariants.

The purpose is not just organization.

The purpose is control over reasoning.

---

# Execution as a Sequence of States

A procedure should not be seen only as a list of actions.

It should be seen as a sequence of increasingly constrained states.

For example:

```text
cart created
→ cart validated
→ total calculated
→ user details validated
→ payment authorized
→ order confirmed
```

Each transition should make the system more precise.

Each transition should preserve validity.

This way of thinking prepares the ground for algorithms, protocols, databases, concurrency and distributed systems.

---

# Abstraction

Abstraction exists because human attention is limited.

A payment component may involve HTTPS, retries, fraud checks, serialization, authentication and external providers.

But the checkout system usually does not need to know those details.

It needs to know whether payment was authorized.

An abstraction does not destroy complexity.

It moves complexity behind a boundary.

A useful abstraction exposes what other parts of the system need in order to reason correctly, while hiding mechanisms that are not relevant at that level.

This connects directly to systems thinking: components matter because of what they own, what they expose, what they depend on and what assumptions they protect.

---

# Core Mental Model

Computational thinking asks:

```text
What is the current state?
What state do we want?
What transformations connect them?
Which states must never exist?
How can the problem be decomposed?
Which details can be hidden safely?
```

These questions are independent of language, framework and technology.

That is why they are durable.

They are the foundation beneath implementation.
