# Module 0.1 — Software Systems Map

> *Modern software is not a program. It is an ecosystem of cooperating systems.*

---

# Objective

Software engineering is often learned one technology at a time.

Networking.

Operating systems.

Databases.

Cloud.

Distributed systems.

Because these subjects are usually introduced independently, it is easy to mistake them for independent fields.

They are not.

They are different perspectives on the same problem: **building software systems that accomplish more than a single program ever could.**

This module is not about learning those disciplines.

It is about understanding **why they exist, how they relate, and where each fits within the larger ecosystem of software systems.**

By the end of this module, you should be able to look at almost any technology and ask:

> *Which problem in the software ecosystem does this solve?*

---

# One Request, Many Systems

Consider an ordinary action.

A user opens an online store and clicks a product.

The browser sends:

```text
GET /products/42
```

It appears to be a single request.

In reality, that request passes through an entire ecosystem of cooperating systems.

```text
                User
                  │
                  ▼
        Browser / Client
                  │
                  ▼
            Networking
                  │
                  ▼
           Application
                  │
        ┌─────────┴─────────┐
        ▼                   ▼
   Business Logic        Storage
        │                   │
        └─────────┬─────────┘
                  ▼
          Operating System
                  │
                  ▼
        Computer Hardware
```

No single component "handles the request."

Each contributes one specific responsibility while depending on the others to fulfill theirs.

The request succeeds not because one system is powerful enough, but because many systems cooperate.

---

# What Is a Software System?

Throughout this curriculum, the word *system* has a precise but practical meaning.

A software system is a collection of **components** that cooperate to achieve a goal.

Each component has a **responsibility**.

Components communicate through **interfaces**.

Together they produce behavior that none of them could produce alone.

A browser is a system.

An operating system is a system.

A database is a system.

Even the complete web application serving our `GET /products/42` request is itself a system composed of many smaller systems.

This idea will appear repeatedly throughout Systems Mastery:

> **Large systems are built by composing smaller systems with clearly defined responsibilities.**

---

# Why Software Became Specialized

Early computers often executed one program at a time.

Modern software rarely does.

A single request may involve persistent data, encrypted communication, multiple processes, remote machines and specialized hardware.

As software became more capable, new engineering problems emerged.

No single abstraction could solve them all.

Instead, engineers developed specialized disciplines, each responsible for solving one class of problems while relying on the solutions provided by others.

The names of those disciplines matter far less than the questions that created them.

---

# The Major Domains

## Computer Architecture

**Question**

> How does hardware execute software?

Every software system ultimately depends on a physical machine.

Processors execute instructions.

Memory stores data.

Hardware defines the capabilities and limits that every higher-level abstraction must respect.

When the server answers `GET /products/42`, every instruction eventually becomes activity inside a processor.

Computer Architecture exists because software cannot be understood independently of the machine that executes it.

---

## Operating Systems

**Question**

> How can many programs safely share one computer?

Once computers began running many programs simultaneously, hardware alone was no longer enough.

Applications needed a coordinator that could allocate resources, isolate failures and allow independent programs to coexist.

The operating system emerged to provide that coordination.

Our server never manipulates hardware directly.

It relies on the operating system to schedule execution, manage memory and access devices safely.

---

## Networking

**Question**

> How can independent computers communicate?

The browser and the server execute on different machines.

Without a common way to exchange information, they might as well exist in different universes.

Networking emerged to allow independent computers to cooperate despite distance, latency and unreliable communication.

Without it, `GET /products/42` never reaches the server.

---

## Storage

**Question**

> How can information survive after a program stops running?

Memory is temporary.

When a process terminates, its state disappears.

Businesses, however, expect customer accounts, inventory and orders to remain available tomorrow.

Storage exists because software requires durable state.

When our application retrieves product information, it depends on data that outlived countless previous executions.

---

## Concurrency

**Question**

> How can multiple activities make progress at the same time?

One request is simple.

Ten thousand simultaneous requests are not.

Servers process many users while performing background work and interacting with storage and networks.

Concurrency emerged because systems needed to coordinate many independent activities without sacrificing correctness.

---

## Distributed Systems

**Question**

> How can multiple computers behave like one system?

Eventually one computer becomes insufficient.

More users arrive.

More data accumulates.

Failures become unavoidable.

Instead of building larger machines forever, software began spreading work across many smaller ones.

Distributed systems emerged to make those independent machines cooperate as though they were parts of a larger whole.

Our single browser request may already involve several servers before the response returns.

---

## Observability

**Question**

> How can we understand what a running system is actually doing?

As systems grew larger, they also became harder to reason about.

When requests become slow or failures appear, intuition is no longer enough.

Observability emerged to make running systems explain themselves through evidence rather than guesswork.

Without it, engineers are forced to diagnose complex systems while effectively blind.

---

## Security

**Question**

> How can systems remain trustworthy?

As software became connected, it also became exposed.

Users expect their identities, data and transactions to remain protected despite mistakes, failures or malicious actors.

Security exists because cooperation without trust is fragile.

Even our simple request may require encrypted communication, authentication and authorization before any product information is returned.

# The Software Systems Map

The domains above are often taught independently.

Real software, however, does not resemble separate chapters in a book.

It resembles an ecosystem.

```text
                              SOFTWARE SYSTEM
                                     │
        ┌────────────────────────────┼────────────────────────────┐
        │                            │                            │
        ▼                            ▼                            ▼
  Networking                    Storage                     Observability
        │                            │
        └──────────────┐       ┌─────┘
                       ▼       ▼
                 Operating System
                       │
             ┌─────────┴─────────┐
             ▼                   ▼
       Concurrency      Computer Architecture
             │                   │
             └─────────┬─────────┘
                       ▼
                    Hardware

Security is not a layer.
It influences every interaction across the ecosystem.

Distributed Systems emerge when Networking,
Storage, Concurrency and Operating Systems
are combined across multiple machines.
```

This map is intentionally **not** a stack.

The relationships matter more than the positions.

Each system depends on others while simultaneously supporting new capabilities above it.

Understanding software therefore means understanding dependencies rather than memorizing layers.

---

# Relationships Matter More Than Layers

A useful way to understand this map is to imagine removing one domain.

Remove networking.

The browser and server can no longer communicate.

The application may still exist, but the request never arrives.

Remove storage.

The server still runs, but every restart erases products, users and orders.

Remove the operating system.

Applications lose the mechanism that coordinates access to hardware.

Nothing above it can execute safely.

The individual systems have not changed.

Their relationships have.

That is the essential idea.

A software system is defined not only by its components, but by the dependencies that allow those components to cooperate.

---

# A Different Way to Think

Beginners often organize knowledge around technologies.

"I'm learning Docker."

"I'm learning Kubernetes."

"I'm learning PostgreSQL."

Experienced engineers tend to organize knowledge around engineering problems.

They ask:

- Which responsibility does this system have?
- Which problem made it necessary?
- Which other systems does it depend on?
- What happens if this system disappears?

Technologies evolve.

The underlying engineering problems remain remarkably stable.

Learning to recognize those problems is the first step toward systems thinking.

---

# Where AI Fits

Artificial intelligence is not outside this map.

An AI inference request still travels through networks.

It still executes inside an operating system.

It still depends on memory, hardware and storage.

Large AI platforms often add distributed execution, but they do not replace the foundations beneath them.

The same ecosystem supports both a web application and an AI service.

Only the workload changes.
